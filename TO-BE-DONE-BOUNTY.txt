[BUG-1] CRITICAL — sendMessage ReferenceError breaks entire chat UI
  File   : c9_jokes/templates/chat.html:925
  Commit : b089fa3
  Detail : sendBtn.addEventListener('click', sendMessage) referenced a function
           that was never defined. The ReferenceError fired inside the IIFE with
           'use strict', halting execution before ALL listeners below line 925
           were attached — quick-prompt buttons, keyboard shortcuts, session
           resume banner, thinking-mode dropdown were all dead.
  Fix    : Replaced with form submit handler (send button is type="submit";
           form.addEventListener('submit',...) already calls send()). Removed
           the redundant extra click listener that caused double-send.
  Test   : Puppeteer E2E 19/19 passed (validate_quick_prompts.mjs).

[BUG-2] CRITICAL — asyncio.wait_for cancels httpx stream iterator (C9)
  File   : c9_jokes/app.py (streaming loop)
  Commit : cfc0496
  Detail : wait_for(line_iter.__anext__(), timeout=15) cancelled the underlying
           coroutine on heartbeat timeout, corrupting the stream iterator state
           and causing empty or truncated responses.
  Fix    : Wrapped with asyncio.shield + asyncio.ensure_future so the read
           coroutine survives the timeout. Added try/finally to cancel the
           background task on all exit paths (disconnect, [DONE], error).
  Test   : tests/test_c9_stream_shield.py — 6 tests, all passing.

[BUG-3] CRITICAL — aiohttp aiodns fails inside Docker (C1 → C3)
  File   : copilot_backend.py (_get_c3_session)
  Commit : cfc0496
  Detail : Default aiohttp TCPConnector uses aiodns (async DNS) which fails
           inside Docker bridge networks, preventing C1 from resolving C3.
  Fix    : Added resolver=aiohttp.ThreadedResolver() + force_close=True to
           TCPConnector to use OS-level DNS and disable stale keepalive.
  Test   : tests/test_c1_threaded_resolver.py — 6 tests, all passing.

[BUG-4] MEDIUM — Race condition creating duplicate aiohttp sessions (C1)
  File   : copilot_backend.py (_get_c3_session_lock)
  Commit : cfc0496
  Detail : Concurrent requests could both read cls._c3_session is None and
           create two separate sessions, leaking the first one.
  Fix    : Added class-level asyncio.Lock (_c3_session_lock); converted
           _get_c3_session to async classmethod that acquires the lock.
  Test   : tests/test_c1_threaded_resolver.py::test_concurrent_calls_produce_single_session
