Tasked operations manual

A practical guide to the Tasked family of C9 pages: how alerts, task definitions, run pipelines, completed runs, and Live Docs fit together. This page is for operators who need to understand what each control does, which API and database rows it touches, and how one page hands context to another.

Mental Model

Task definition -> run -> step results -> alert/completion

The Tasked pages are one orchestration surface split across five views. Tasked is where a user creates, edits, clones, and starts a task. Pipeline follows an execution as it moves through workflow steps. Alerts shows operational signals created by matching conditions. Completed lists terminal runs for review. Live Docs keeps seeded trace examples that prove the system can reproduce known positive and negative paths.

1. AuthorCreate or load a template in Tasked. Store task fields, template data, and steps.
2. ExecuteStart a manual, recurring, or continuous run. Each run gets a new run ID.
3. TracePipeline reads the run, step results, events, feedback, and alerts as one trace.
4. AlertAlert steps create task_alerts only when the trigger logic matches.
5. CloseCompleted reviews terminal runs and gives redo, clone, archive, and preview links.

Shared Architecture

FrontendEach page extends base.html, uses the shared navbar, and fetches JSON APIs with page-local JavaScript.
BackendFastAPI routes render templates and expose JSON endpoints under /api/*.
DatabaseSQLite stores task definitions, templates, workflow steps, runs, step results, alerts, events, feedback, and recovery sessions.
InteractionTasked creates or runs data. Pipeline and Completed read run history. Alerts reads alert rows. Live Docs seeds and validates known tasks.
Important rule: a saved Tasked row is the source of truth. Templates are only starting points. After a template is loaded, edited, and saved, the saved row controls future runs.

Tasked - Builder and Orchestrator

Open /tasked

Tasked is the doer and source of truth. It stores the task definition, schedule, workflow steps, template parameters, prompt/command text, alert policy, completion policy, and runtime launch path.

Primary UI knobs
  • Editable Templates: load, save, clone, or archive DB-backed templates.
  • Tasked Chat Planner: type a plain-English request, optionally refine with the M365 agent, then apply the draft.
  • Template parameters: weather city/temperature, distance from/to/threshold, and combo chain items.
  • Core fields: name, mode, output type, schedule, interval minutes, tabs required, and active state.
  • Workflow steps: trigger, condition, chat, sandbox, agent, multi-agent, multi-agento, alert, complete.
  • Actions: Save Tasked, Start Task, Run Again, Clone/Edit, Pause, Resume, Restart, Archive, bulk actions.
How to operate
  • Load a template when the use case matches an existing pattern.
  • Edit structured knobs before editing raw prompts. Structured knobs keep steps and prompts synced.
  • Use combo mode for multiple templates; choose AND, OR, or NOR before saving.
  • Click Save Tasked to persist the row, then Start Task or Run Again to create a new run.
  • Use the trace panel links to jump into Pipeline, Alerts, Completed, or Preview.
Backend and APIs
  • GET /tasked renders the builder.
  • GET/POST /api/tasks lists and saves task definitions.
  • POST /api/tasks/{id}/run executes a task.
  • POST /api/tasks/draft-from-text creates a draft from plain English.
  • GET/POST /api/task-templates manages editable templates.
  • GET /api/task-runs feeds recent run history.
Database touchpoints
  • task_definitions: saved Tasked rows.
  • task_templates: editable template catalog.
  • task_workflow_steps: ordered workflow steps.
  • task_runs: execution records.
  • task_step_results: per-step output and parsed JSON.

Alerts - Operational Signal Board

Open /alerts

Alerts shows signals created by Tasked alert steps. An alert is not a task itself; it is evidence that a run met a trigger condition.

UI knobsStatus filter, severity filter, run-status filter, search, refresh interval, card/table view, group-by-status, bulk selection, detail panel.
ActionsAcknowledge, resolve, snooze, reopen, bulk acknowledge, bulk resolve, open Tasked, open Pipeline, and open Preview.
APIsGET /api/alerts, POST /api/alerts/{id}/status, and POST /api/alerts/{id}/ack.
DB and joinsReads task_alerts and joins task/run context from task_definitions and task_runs.

Pipeline - Run Trace Monitor

Open /piplinetask

Pipeline is run-centric. It assembles one view from task definition, selected run, all runs for that task, alerts, events, step results, feedback, recovery sessions, and synthesized timeline data.

UI knobsTask ID filter, run/status filtering through query/API, refresh controls, live card state, step strip, feedback panels, and timeline.
When to use itUse Pipeline when a task is running, failed, skipped an alert, or produced a surprising result. It shows the step that made the decision.
APIsGET /api/task-pipelines?task_id=..., ?run_id=..., ?status=....
DB and backendBuilds a composed payload from task_runs, task_step_results, task_alerts, task_events, and feedback rows.

Completed - Terminal Run Review

Open /task-completed

Completed lists terminal runs only: completed, failed, and cancelled. It is for review, exporting, and choosing what to do next with the task definition.

UI knobsStatus selector, task ID filter, refresh interval, summary cards, duration chart, JSON/CSV export, and task action buttons.
ActionsRedo creates another run on the same trace. Clone Task creates a new editable task. Archive hides the saved task from normal lists.
APIsGET /api/task-completed, plus task lifecycle APIs such as /api/tasks/{id}/redo, /clone, and /archive.
DB and meaningReads terminal rows from task_runs, then enriches each item with task metadata, step results, latest alert, and recovery context.

Live Docs - Seeded Regression Lab

Open /tasked-live-doc

Live Docs is both documentation and a controlled test bench. It renders legacy examples plus DB-backed template trace tasks such as TRACE-200, TRACE-300, TRACE-400, TRACE-500, and TRACE-800 custom aggregate workflows.

UI knobsDocs tab, Live tab, Run, Run All, Validate, Validate All, auto-refresh, trace cards, and right-side detail panel.
How it worksOn load it seeds stable Tasked rows for template traces, then reads those same task IDs and run IDs back into the page.
APIsGET /api/tasked-live-doc/traces, task run APIs, task preview API, Pipeline API, Alerts API, and Completed API.
DB roleUses normal Tasked tables. Live Docs is not a separate engine; it proves the regular Tasked engine through known trace rows.

Operating Playbooks

Create and run a taskOpen Tasked, load a template or draft from text, edit structured knobs, save, start, then use Pipeline to monitor.
Investigate an unexpected resultOpen Pipeline for the task ID, inspect step results and parsed JSON, then open Preview or Completed for the terminal summary.
Handle an alertOpen Alerts, filter by open/critical/run status, inspect the detail panel, then acknowledge, snooze, resolve, or jump back to Tasked/Pipeline.
Repeat without changing traceUse Run Again or Redo. The Task ID stays the same, but the run ID and execution number change.
Edit safelyUse Clone/Edit when you need a new trace. Use structured template controls before editing raw prompts directly.
Validate examplesUse Live Docs to run positive and negative template traces before trusting a new template pattern.

Runtime Dependencies and 6080

Chat-backed Tasked runs depend on the C1 and C3 path. C3 browser-auth exposes the noVNC/sign-in surface on port 6080, while the C9 status and auth pages summarize session state inside port 6090. If M365 is expired or C3 has no ready tabs, Tasked chat runs can fail, stall, or require re-authentication.
