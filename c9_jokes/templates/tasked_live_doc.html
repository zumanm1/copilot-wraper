📋 Tasked Live Documentation — create · run · trace · validate · 5 types · 6 traces C9b:6090 → C1b:8000 (chat) · C12b:8210 (sandbox)
How the System Works
1. task_id
Created when you POST /api/tasks. Permanent anchor — never changes across all runs, alerts and steps.
2. run_id
Created on each POST /api/tasks/{id}/run. Each execution gets a unique trun_xxxxxxxx.
3. alert_id
Integer created when the alert step fires. Only created if trigger_mode=always or JSON returns triggered:true.
4. step_id
Each step in the workflow has a fixed ID (e.g. t1_chat). Results stored per step per run.
Universal 4-Step Workflow (all task types)
Step 1 · trigger
Start
Records timestamp, schedule kind, marks run as queued→running. No external call.
Step 2 · chat or sandbox
Execute
chat: POST C1b:8000/v1/chat/completions → LLM text
sandbox: POST C12b:8210/exec → stdout/exit_code
Step 3 · alert
Fire Alert
always: alert fires unconditionally
json: fires only if output contains {"triggered":true}
Step 4 · complete
Close
Sets terminal_reason=workflow-complete, status=completed. Run is final.

⚡
alert Alert Task
Evaluates a condition — fires alert only when triggered
How created POST /api/tasks · tasked_type="alert" · trigger_mode="json" How triggered Manual run. LLM must return JSON: {"triggered":true,"reason":"..."} to fire alert. Execution C9b sends planner question to C1b. Response parsed for "triggered":true field. Alert fires Conditional — only when JSON response contains triggered:true. severity=warning typically. Completion Same: terminal_reason=workflow-complete · status=completed Key difference trigger_mode="json" vs "always". Use for threshold checks, anomaly detection.
planner asks condition
→
C1b evaluates
→
parse {"triggered":true}
→
alert fired if true
# planner_prompt forces JSON response "planner_prompt": "Is CPU 95% > threshold 80%? Respond ONLY as JSON: {\"triggered\":true,\"reason\":\"...\"}" "trigger_mode": "json" ← key setting

action Action Task
Runs shell commands on C12b sandbox — returns raw stdout
How created POST /api/tasks · tasked_type="action" · mode="sandbox" · executor_target="C12b" How triggered Manual run. C9b calls POST C12b:8210/exec with the command string. Execution C12b runs the shell command in /workspace. Returns stdout, stderr, exit_code, session_id. exit_code≠0 → failed. C12b response {"stdout":"...","stderr":"","exit_code":0,"session_id":"c12b_xxx"} Validation Optional validation_command and test_command also sent to C12b separately. Completion Execution:completed · Validation:completed · Test:completed

🔗
hook Hook Task
Generates webhook payloads — triggers external systems
How created POST /api/tasks · tasked_type="hook" · mode="chat" How triggered Manual run. Planner instructs LLM to generate a webhook JSON payload. Execution C9b calls C1b with hook-specific prompt. LLM generates JSON payload (e.g. Slack, GitHub, CI/CD webhook body). Alert fires Yes — trigger_mode=always. Alert confirms the hook was dispatched. Completion terminal_reason=workflow-complete · output contains webhook JSON Extension Payload from output can be forwarded to external URLs in future sandbox steps.
planner: "generate webhook"
→
C1b generates JSON payload
→
alert: hook dispatched
→
complete
# Hook planner instructs LLM to act as dispatcher "planner_prompt": "Generate a Slack webhook payload for a build-success notification." "executor_prompt": "You are a CI/CD dispatcher. Produce valid Slack webhook JSON." # Output: {"text":"Build passed ✅","attachments":[...]}

combined Combined Task
All types in one run: output + alert + action + hook paths exercised
How created POST /api/tasks · tasked_type="combined" How triggered Manual run. Planner requests a structured multi-section report. Execution C1b LLM generates structured output (SUMMARY + ACTION + SEVERITY). All paths covered. Alert fires Always — severity=high. Combined tasks typically produce critical-level alerts. Completion Longest running type due to structured full-context LLM generation (7m 40s observed).

Label each section clearly. FINDINGS / RISK_LEVEL / REMEDIATION." # severity=high forces priority alert "alert_policy": {"severity":"high"}
