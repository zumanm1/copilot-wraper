How IDs Work
task_id
Created when you call POST /api/tasks.
Format: task_xxxxxxxx (8 hex chars).
Permanent anchor — never changes across all runs, alerts, and steps. Shown in Tasked, Piplinetask, Alerts, Completed, and Preview pages.
run_id
Created on each POST /api/tasks/{id}/run.
Format: trun_xxxxxxxx.
One per execution. A single task can have many runs. Stored in task_runs table.
alert_id
Integer. Created only when the alert workflow step fires.
Fires if trigger_mode=always, or if the LLM response contains {"triggered":true} when trigger_mode=json.
Stored in task_alerts table.
step_id
Fixed per task (e.g. nt1_trigger, nt1_chat).
Defined when the task is created. Step results are stored per step per run in task_step_results. Visible in the Piplinetask step diagram.
Universal 4-Step Workflow
Step 1 · trigger
Start
Records timestamp and schedule kind. Transitions run from queued to running. No external API call.
Step 2 · chat or sandbox
Execute
chat: POST C1b:8000/v1/chat/completions → LLM text response
sandbox: POST C12b:8210/exec → stdout / exit_code / session_id
Step 3 · alert
Fire Alert
always: alert fires unconditionally
json: fires only when output contains {"triggered":true}
Step 4 · complete
Close
Sets terminal_reason=workflow-complete and status=completed. Run is final.

alert Alert Task
Evaluates a condition — fires alert only when triggered
How created POST /api/tasks · tasked_type="alert" · trigger_mode="json" How triggered Manual run. LLM must return JSON with {"triggered":true,"reason":"..."} to fire alert. Execution C9b → C1b with condition-evaluation prompt. Response parsed for triggered field. Key difference trigger_mode="json" vs "always". Use for threshold checks, anomaly detection. Completion terminal_reason=workflow-complete · status=completed
planner: ask condition→ C1b evaluates→ parse triggered:true→ conditional alert
# Planner forces JSON-only response "planner_prompt": "Is CPU 95% above threshold 80%? Respond ONLY as JSON: {\"triggered\":true,\"reason\":\"...\"}" "trigger_mode": "json" <-- key setting
TRACE-002 · task_adc06130 · trun_eae69a71 · alert=432
action Action Task C12b
Runs shell commands on C12b sandbox — returns raw stdout
How created POST /api/tasks · tasked_type="action" · mode="sandbox" How triggered Manual run. C9b calls POST C12b:8210/exec with command string. C12b response {"stdout":"...","stderr":"","exit_code":0,"session_id":"c12b_xxx"} Failure condition exit_code != 0 marks step as failed. stderr captured separately. Validation Optional validation_command + test_command also sent to C12b.
mode=sandbox→ POST C12b:8210/exec→ stdout exit_code=0→ alert + complete
# Direct C12b call (what C9b does internally) curl -X POST http://localhost:8210/exec \ -d '{"command":"df -h /workspace && echo DONE"}' # <-- {"stdout":"...","exit_code":0,"session_id":"c12b_xxx"} # In a C9 task definition: "mode": "sandbox", "executor_target": "C12b" "steps": [{"kind":"sandbox","config":{"command":"df -h"}}]
TRACE-003 · task_f0642138 / TRACE-010 · task_fe608355
hook Hook Task
Generates webhook payloads — triggers external systems
How created POST /api/tasks · tasked_type="hook" · mode="chat" How triggered Manual run. Planner instructs LLM to generate a webhook JSON payload. Execution C9b → C1b with hook-specific prompt. LLM generates JSON payload (Slack, GitHub, CI/CD). Alert fires Yes — trigger_mode=always. Alert confirms hook was dispatched. Extension Payload from output can be forwarded to external URLs in future sandbox steps.
planner: generate webhook→ C1b generates JSON→ alert: dispatched→ complete
# Planner instructs LLM to act as dispatcher "planner_prompt": "Generate a Slack webhook payload for a build-success notification." "executor_prompt": "You are a CI/CD dispatcher. Produce valid Slack webhook JSON." # Output: {"text":"Build passed","attachments":[...]}
TRACE-004 · task_f7632723 · trun_f0c82c94 · alert=435
combined Combined Task
All type paths exercised: output + alert + action + hook in one run
How created POST /api/tasks · tasked_type="combined" How triggered Manual run. Planner requests a structured multi-section report. Execution C1b LLM generates structured output (SUMMARY + ACTION + SEVERITY). All paths covered. Alert fires Always — severity=high. Combined tasks produce critical-level alerts. Duration Longest of all types — observed 7m 40s for full structured LLM generation.
multi-section prompt→ C1b full report→ severity=high alert→ completed
# Multi-section prompt covers all paths "planner_prompt": "Produce a security audit covering FINDINGS, RISK_LEVEL, and REMEDIATION_STEPS." "executor_prompt": "You are a security auditor. Label each section: FINDINGS / RISK_LEVEL / REMEDIATION." "alert_policy": {"severity":"high"} <-- priority alert
